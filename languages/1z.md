# 1z Standards

## Naming

- Words: `kebab-case` (`parse-token`, `make-person`, `assert-true`)
- 1z is NOT Forth -- definitions use `name: [ body ] ;`, not `: name body ;`

### Prefix and Suffix Conventions

| Prefix/Suffix | Meaning | Examples |
|---|---|---|
| `name:` | Symbol literal | `foo:`, `name:`, `+:` |
| `#name` | Sequence operation | `#len`, `#map`, `#filter`, `#nth` |
| `@name` | Associative operation | `@get`, `@set`, `@has?`, `@keys` |
| `?name` | Option/result combinator | `?map`, `?and-then`, `?or-else` |
| `>name` | Conversion (target type) | `>float`, `>string`, `>iterator` |
| `<name>` | Smart constructor (validates inputs) | `<range>`, `<ndarray>`, `<channel>` |
| `name?` | Predicate | `fixnum?`, `empty?`, `color:red?` |
| `name!` | Mutating/dangerous | `@set!`, `#push!`, `#nth!` |
| `>>field` | Struct setter | `>>name`, `>>age` |
| `field>>` | Struct getter | `name>>`, `age>>` |
| `make-name` | Raw positional constructor | `make-person`, `make-range` |
| `unmake-name` | Positional destructuring | `unmake-person`, `unmake-range` |
| `name(variant)` | Variant/parameterized word | `>float(exact)`, `vector(fixnum)` |
| `parse-name` | Parse string into value | `parse-json`, `parse-integer` |
| `emit-name` | Serialize value into string | `emit-json`, `emit-json-pretty` |

### Anti-Stutter Rule

Words in a library file should not repeat the module name as a prefix. A word in `lib/foo.1z` should be `bar`, not `foo-bar`, since qualified access provides `foo.bar`. Exceptions:

- Common or ambiguous words where the unprefixed name conflicts with a well-known semantic (e.g., `channel-each` vs `each`)
- Conversion words that keep the source type for clarity (e.g., `channel>array` not bare `>array`)
- Higher-level wrappers around same-named native primitives (e.g., `tcp-listen` wraps native `listen`)
- Only exact module-name prefixes count as stuttering (`serve-forever` in `server.1z` is fine because `serve-` is not `server-`)

## Comments

- Line comments: `\` (backslash followed by space)
- Doc comments: `\\` (double backslash), attach to the next word definition
- `( ... )` is stack effect notation, NOT a comment

```
\ this is a line comment
\\ This is a doc comment for the next word.
double: ( n -- n ) [ 2 * ] ;
```

## Syntax

Word definitions: a symbol (name with trailing `:`), optional stack effect, optional markers, a quotation body in `[ ]`, and `;` to finalize:

```
double: ( n -- n ) [ 2 * ] ;
greet: [ "hello" print-line ] ;
PI: const [ 3.14159 ] ;
```

Stack effects are declared with `( inputs -- outputs )` between the name and body. They are optional but recommended for public words.

Quotations (`[ ... ]`) are first-class values. They are the basis for control flow, word bodies, and higher-order programming:

```
[ 2 * ] call           \ execute a quotation
[ . ] 10 times         \ loop combinator
0 > [ "yes" ] when     \ conditional
```

## Project Structure

Simple standalone projects use a flat layout:

```
project/
├── main.1z            Entry point
├── lib/               Library modules
│   ├── foo.1z
│   └── bar.1z
└── tests/
    └── integration/   Integration test files
```

### Imports

Load a module with `use`:

```
use "testing" ;              \ standard library
use "math" ;                 \ lib/math.1z
use "./local-file.1z" ;     \ relative path
```

Selective imports restrict which words are brought into scope:

```
use { assert= assert-true } "testing" ;
```

Qualified access uses dot notation after loading:

```
math.sin math.cos math.pi
```

`use` imports words into scope but does not create a module binding.
For qualified access, bind the module with `load`:

```
math: "math" load ;
math.sin math.cos math.pi
```

## Error Handling

- `recover` catches errors from a quotation: `[ risky ] [ handler ] recover`
- `ignore-errors` silently drops errors: `[ risky ] ignore-errors`
- `throw` raises an error: `f "message" EUserThrown make-error throw`
- `try` wraps the result: `[ risky ] try` returns `result:ok` or `result:err`
- Error fields are accessed via `@get`: `err error-type: @get`
- Use smart constructors (`<name>`) to validate inputs at construction time

## Idioms

- Prefer quotation-based combinators (`dip`, `keep`, `bi`) over local definitions for stack manipulation
- Use `#collect` to materialize lazy sequences when a concrete collection is needed; higher-order sequence words (`#map`, `#filter`) return lazy iterators by default
- Prefer `method{` dispatch over conditional type checks (`dup fixnum? [ ... ] [ ... ] if`)
- Use virtual types (`virtual{ ... }`) as newtypes to create distinct type identities
- Use `enum{` for sum types; use `match{` for exhaustive dispatch with parse-time validation
- Truthiness: only `f` is falsy; `0`, `""`, `{ }`, and `nil` are all truthy

## Testing

Import the testing library:

```
use "testing" ;
```

### Assertion Words

| Word | Stack Effect | Purpose |
|---|---|---|
| `assert=` | `( actual expected msg -- )` | Check equality |
| `assert-true` | `( bool msg -- )` | Check value is true |
| `assert-false` | `( bool msg -- )` | Check value is false |
| `assert-error` | `( quot msg -- )` | Check quotation throws any error |
| `assert-error-type` | `( quot expected-type msg -- )` | Check quotation throws specific error type |
| `test-section` | `( msg -- )` | Print a section header |

### Test Style

- Prefer assertion words over printing values and relying on golden file comparison
- Use `test-section` to group related assertions
- Name tests descriptively: the message string should explain what is being tested

```
use "testing" ;

"arithmetic" test-section
4 2 2 + "two plus two" assert=
t 1 0 > "one is greater than zero" assert-true

"error handling" test-section
[ 1 0 / ] "division by zero throws" assert-error
[ 1 0 / ] division-by-zero: "division by zero type" assert-error-type
```

### Integration Tests

- Place integration tests in `tests/integration/`
- Each test is a `.1z` file with optional companion files:
  - `.stdin` for input
  - `.stdout.golden` and `.stderr.golden` for expected output
  - `.flags` for per-test CLI flags
  - `.exitcode` for expected exit codes (default: 0)

### Golden File Workflow

1. Write or update the `.1z` test file
2. Run `make update-golden` to regenerate golden files
3. Review the diffs to confirm they match expectations
4. Validate no `FAIL:` appears in golden files
